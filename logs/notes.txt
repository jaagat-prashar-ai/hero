WDS build job (build-wds-parallel, the data-sharding/moving job) — No, there's no shared temp folder. Each rank is a fully independent Lilypad job submission (potentially on a different physical node). Each one downloads its own assigned slice of clips straight from HuggingFace, writes shards to its own local/ephemeral tempfile (S3ShardWriter._open_shard in build_webdataset.py, defaults to that pod's local /tmp), and uploads directly to S3. There's no cross-rank read/write of local files — by design, ranks never need to see each other's data since the only handoff point is S3.

Training job (masking-cot-cluster, cluster.yaml) — Yes, but only incidentally. local_data_dir: /mnt/work/tmp/wds_cache is shared across all 8 ranks — run.py's _acquire_shards() has rank 0 download the WDS shards from S3 into that path, and ranks 1-7 just poll the same path waiting for files to appear (masking/training/run.py:137-143). That only works because cluster_resources.gpu_machine_type: a100.8 + num_gpus: 8 means all 8 ranks run as separate processes on one single 8-GPU physical node — so /mnt/work is trivially the same local filesystem for all of them, not a deliberately-provisioned shared network volume. If this job were ever split across multiple nodes, that polling scheme would break unless /mnt/work is also genuinely network-mounted cluster-wide — I can't confirm that from the code alone, only that the current single-node config makes it work either way.

So: shared-folder coordination exists only in the training job between GPU ranks on the same node, not in the data-build/sharding job.
